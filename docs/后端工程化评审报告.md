## 一、架构设计评审

### 1.1 整体架构评分：8/10

**架构图**:
```
┌─────────────────────────────────────────────────────────────┐
│                        Controller 层                        │
│  WireController → PurchaseController → AppointmentController  │
│  (每个模块独立 Controller，URL 前缀 /api/{module})           │
├─────────────────────────────────────────────────────────────┤
│                        Service 层                           │
│  WireServiceImpl → PurchaseServiceImpl → AppointmentServiceImpl│
│  (接口 + 实现分离，@RequiredArgsConstructor 构造器注入)       │
├─────────────────────────────────────────────────────────────┤
│                        Mapper 层                            │
│  WireMapper → PurchaseMapper → AppointmentMapper             │
│  (MyBatis Plus BaseMapper，LambdaQueryWrapper 类型安全)      │
├─────────────────────────────────────────────────────────────┤
│                        Entity 层                            │
│  Wire → Purchase → Appointment                               │
│  (@TableName 映射，@TableField 审计字段)                      │
├─────────────────────────────────────────────────────────────┤
│                    公共模块 (common/)                         │
│  Result<T> / BusinessException / GlobalExceptionHandler      │
│  JwtUtils / OperationLogAspect                               │
├─────────────────────────────────────────────────────────────┤
│                    配置模块 (config/)                         │
│  MybatisPlusConfig / CorsConfig / SecurityConfig              │
└─────────────────────────────────────────────────────────────┘
```

**亮点**:
- ✅ 按领域分包（modules/{domain}/），模块边界清晰
- ✅ Controller → Service → Mapper → Entity 四层分层，职责单一
- ✅ 接口 + 实现分离（XxxService / XxxServiceImpl），便于测试和替换
- ✅ 统一返回格式 `Result<T>`，前端对接一致
- ✅ 全局异常处理 + AOP 操作日志

**改进建议**:
- ⚠️ **缺少 DTO 层**: 当前 Entity 直接暴露给 Controller，建议增加 DTO 层隔离内外模型
- ⚠️ **缺少 VO 层**: 查询响应直接使用 Entity，建议增加 VO 层定制返回字段
- ⚠️ **缺少 Facade 层**: 跨模块调用（如 Points 调用 Member）直接注入 Service，建议增加 Facade 层

## 三、性能评审

### 3.1 数据库性能评分：7.5/10

**亮点**:
- ✅ 已添加复合索引（idx_member_time、idx_member_date 等）
- ✅ 使用 MyBatis Plus 分页插件（PaginationInnerInterceptor）
- ✅ 软删除使用 `deleted` 字段，避免物理删除

**问题**:

| 问题 | 影响 | 严重性 |
|------|------|--------|
| **N+1 查询**: `getPointsBalance` 先查 Member 再返回，每次调用都查数据库 | 中 | 🟡 |
| **全表扫描风险**: `getLowStockItems` 使用 `le(currentStock, minStock)`，无法使用索引 | 中 | 🟡 |
| **大事务风险**: `ExchangeServiceImpl.redeemItem` 包含积分扣减 + 库存扣减 + 记录创建，事务时间较长 | 高 | 🔴 |
| **缺少缓存**: 线材字典、积分规则等热点数据未缓存 | 中 | 🟡 |
| **批量操作缺失**: `generateQRCodes` 逐条插入 N 次，应使用批量插入 | 中 | 🟡 |

**建议**:
```java
// 1. 使用缓存
@Cacheable(value = "wire", key = "#id")
public Wire getWireById(Long id) { ... }

// 2. 批量插入
@Insert("<script>INSERT INTO tb_qr_code (...) VALUES " +
        "<foreach collection='codes' item='c' separator=','>" +
        "(#{c.qrCode}, #{c.wireId}, ...)" +
        "</foreach></script>")
void batchInsertQRCodes(@Param("codes") List<QRCode> codes);

// 3. 大事务拆分
@Transactional(rollbackFor = Exception.class)
public void redeemItem(...) {
    // 仅执行写操作，查询移到事务外
}
```

### 3.2 并发性能评分：7/10

**亮点**:
- ✅ 积分操作使用原子 SQL 更新
- ✅ 库存扣减使用原子 SQL 更新
- ✅ 使用数据库事务保证一致性

**问题**:

| 问题 | 影响 | 严重性 |
|------|------|--------|
| **库存同步延迟**: 库存更新依赖 AOP 或手动调用，可能遗漏 | 高 | 🔴 |
| **预约冲突**: 同一技师同一时段可被多人预约 | 高 | 🔴 |

```java
// 1. 预约冲突检测
@Transactional
public void createAppointment(Appointment appointment) {

        new LambdaQueryWrapper<Appointment>()
            .eq(Appointment::getTechnicianId, appointment.getTechnicianId())
            .eq(Appointment::getAppointmentDate, appointment.getAppointmentDate())
            .eq(Appointment::getTimeSlot, appointment.getTimeSlot())
            .ne(Appointment::getStatus, "cancelled")
    );
    if (count > 0) throw new BusinessException("该时段已被预约");
    // ...
}

// 2. 积分余额缓存
@Cacheable(value = "points_balance", key = "#memberId")
public Integer getPointsBalance(String memberId) { ... }
```

### 4.1 安全评分：7.5/10

- ✅ 角色权限控制（ADMIN / TECHNICIAN）
- ✅ 跨域配置（CorsConfig）
- ✅ 软删除防止数据丢失

**问题**:

| 问题 | 影响 | 严重性 |
|------|------|--------|
| **JWT 密钥管理**: 密钥存储在 application.yml，未使用配置中心 | 高 | 🔴 |
| **Token 无失效机制**: JWT 过期后无法主动失效（无黑名单） | 中 | 🟡 |
| **敏感数据明文**: 手机号等敏感数据未加密存储 | 中 | 🟡 |
| **SQL 注入风险**: 使用 `${}` 拼接 SQL（如 `setSql("points = points + " + points)`） | 高 | 🔴 |
| **日志敏感信息**: 操作日志可能记录敏感信息（如密码） | 中 | 🟡 |

**建议**:
```java
// 1. 使用 #{} 参数化，避免 SQL 注入
// 错误: .setSql("points = points + " + points)
// 正确: 使用 @Update 注解或 XML 映射
@Update("UPDATE tb_member SET points = points + #{points} WHERE member_id = #{memberId}")
int addPointsAtomic(@Param("memberId") String memberId, @Param("points") Integer points);

// 2. 敏感数据加密存储
@TableField(typeHandler = AesTypeHandler.class)
private String phone;

// 3. Token 黑名单（使用 Redis）
@Cacheable(value = "token_blacklist", key = "#token")
public boolean isTokenBlacklisted(String token) { ... }
```

## 九、总结

本工程是一个**结构清晰、代码规范、功能完善**的中型 Spring Boot 项目，适合单店铺体育器材服务场景。

**核心优势**:
- 领域驱动设计，模块边界清晰
- MyBatis Plus 最佳实践，代码简洁
- 统一异常处理和返回格式
- AOP 操作日志，审计追踪完善

**主要风险**:
- SQL 注入和预约冲突是必须修复的安全和一致性问题
- 缺少缓存和异步处理，高并发场景下性能受限
- 不支持多店铺，扩展性受限

**建议**: 优先修复 P0 问题（2-3 天），再逐步完善 P1/P2 改进项。
